# Database (PostgreSQL)

**Version**: 16-alpine | **Port**: 5432 | **Container**: hermes-postgres

## Setup

Single shared PostgreSQL instance with per-service databases, initialized by `hermes-be/infra/postgres/init.sql`:
- `hermes_auth`
- `hermes_user`
- `hermes_guild`
- `hermes_channel`
- `hermes_chat`
- (others as services are implemented)

No cross-service foreign keys — bounded contexts at the storage layer.

## Migrations

Per-service, managed by SQLx:

```
services/auth-service/migrations/
services/user-service/migrations/
services/guild-service/migrations/
services/channel-service/migrations/
```

Migrations run automatically on service startup via `sqlx::migrate!()`.

## SQLx Compile-Time Checking

All queries use `sqlx::query!()` / `query_as!()` which verify SQL against the live schema at compile time. Requires either:
- Live DB during development, OR
- Checked-in `.sqlx/` directory (generated by `make sqlx-prepare`) for offline/CI builds

## Test Database

Integration tests use `testcontainers` to spin up fresh Postgres instances. Each service's `tests/common/setup.rs` manually lists all migrations to run.

**Important**: When adding a new migration, it must be added to both the `migrations/` directory AND the `setup.rs` constants list.

## Post-MVP Plan

Split to database-per-service with fully isolated instances. Scaling path: PgBouncer → read replicas → sharding by `guild_id` or `user_id`.
